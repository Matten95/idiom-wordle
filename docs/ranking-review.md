# Wordle 排行榜机制审查

更新时间: 2026-07-10

结论: 排行榜两个 P0 已在代码层修复，完成部署清单后可形成“可信今日榜”；周榜、总榜和聚合统计等 P1 仍未完成。当前代码的今日榜完成度约 78/100；若按跨日累计的全局榜标准计算，完成度约 35/100。

## 已完成能力

- 使用服务端 `openid` 识别玩家，不向榜单客户端返回 `openid`。
- 游戏日统一为北京时间，服务端会校验日期、每日答案和 1-6 次范围。
- 同一玩家同一天优先保留更好的获胜成绩。
- 今日榜按胜负、猜测次数、完成时间排序。
- 结算页已有今日人数、胜率、猜次分布和“击败约多少玩家”。

## 主要问题

### P0 当日答案被排行榜提前泄露

状态：代码已修复，等待重新部署 `getRanking` 和真机验收。

`getRanking` 不再返回 `answerText`；今日榜页不再展示答案，也不再用本地历史冒充全网排行。进行中的可信会话只返回题号和公开部首，答案仅在当前玩家结算后返回。

### P0 成绩仍可伪造

状态：代码已修复，等待配置 `DAILY_GAME_SECRET`、创建 `daily_game_sessions` 并重新部署 `submitResult`。

旧的 `won/attempts/answerText` 直传接口会返回 `LEGACY_SUBMISSION_REJECTED`。主局现在由服务端开局、逐次判色和结算，数据库事务原子更新每日会话与 `verified: true` 成绩；答案使用仅云函数可见的密钥洗牌，客户端公开词库无法推算可信每日答案。

### P1 不是跨日全局排行榜

页面始终传入今日日期。云函数即便不传日期，也只是把每天的记录混排，同一玩家会出现多次；没有累计胜场、平均猜次、纯净连胜、周榜或总榜。

建议：增加按 `openid` 聚合的玩家统计表，提供今日榜、周榜、总榜，并返回当前玩家自己的真实排名。

### P1 统计最多读取 1000 条

今日总人数、胜率和猜次分布通过 `.limit(1000)` 在内存计算。玩家超过 1000 后，结算页“击败百分比”和胜率会静默失真。

建议：使用数据库聚合或分桶计数表；成绩写入时原子更新每日统计，不在每次读取时扫描记录。

### P1 并发可能产生重复成绩

成绩写入采用“先查询、再新增/更新”，没有事务或唯一约束。两个并发请求都可能认为记录不存在，从而写出同一玩家同一天的重复记录。

建议：使用确定性文档 ID，例如 `openid_date`，或事务/唯一索引保证幂等。

### P1 云端空榜与本地历史混用

云端无记录或调用失败时，页面把玩家自己的跨日历史包装成“今日排行”，但不展示日期。这会让单人历史看起来像全网榜单。

建议：页面明确拆成“今日榜”和“我的印记”；云端失败只隐藏今日榜，不把本地历史冒充全网数据。

### P2 排名与输入边界不完整

- `limit` 没有服务端钳制，也没有分页。
- 返回的 `total` 是当前页长度，不是全榜人数。
- 失败成绩允许 1-6 次，理论上可提交“一次就失败”的异常记录。
- 昵称和头像由客户端提交，缺少长度、字符和资源地址校验。
- 正式查询需要配置 `date + won + attempts + updatedAt` 复合索引，否则云数据库可能拒绝排序。
