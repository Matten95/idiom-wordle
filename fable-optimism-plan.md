## 一、项目现状认知

### 1.1 产品形态

「成语日课」是一款新中式视觉的成语猜词微信小程序,包含四个玩法闭环:

| 玩法 | 页面 | 形态 |
|---|---|---|
| 每日部首猜词(主局) | `pages/index` | Wordle 式 6 次机会,拼音五色反馈 + 部首提示,每日全网同题 |
| 提示猜词(单人) | `pages/idiom-match` | 5 条递进线索限时翻牌,百分制计分,激励广告解锁「锦囊」 |
| 好友对局(双人) | `pages/room-lobby` + `pages/two-player-game` | 房间码/分享深链进房,提示者-猜词者角色互换,云数据库实时同步 |
| 自由练习 | `pages/index?mode=practice` | 随机大众题,不计连胜 |

配套:历史印记(排行/记录)、我的印记册(资料 + 统计)。

### 1.2 技术形态

- **前端**:原生 WXML/WXSS/JS,无自定义组件,无 TypeScript;页面用自绘 tab-bar + `wx.reLaunch` 模拟 Tab 切换
- **后端**:微信云开发,独立云函数(`getDailyIdiom` / `submitResult` / `getRanking` / `manageRoom` / `submitHint` / `submitGuess` / `getRoomState`),云数据库 3 个集合(`idiom_rooms`、`idiom_hints_live`、`game_results`)
- **数据**:`data/idioms.json` 单一词库源(120 条,Lv1-4),`scripts/sync-data.js` 派生复制到小程序端与 2 个云函数;提示题库 320 条(200 精选 + 120 派生)
- **质量**:`test/project-health.js` 518 行健康检查(字符串断言为主),引擎冒烟测试 + 365 天选题回归;当前全部通过
- **关键降级设计**:云不可用回退本地、`.watch()` 失败回退 2s 轮询、无 AppID/广告位未配置绕过广告 SDK——韧性意识良好

### 1.3 综合判断

产品打磨程度明显高于同阶段独立小程序(PM 复评 92 分有据可依),文案、视觉语言(印章/收桌/开局令)统一且有辨识度。**当前真正的瓶颈不在"再打磨一轮 UI",而在:内容供给量、留存/传播机制缺失、公平性与数据一致性隐患、以及上线前的可观测性。**

## 二、资深产品专家视角

### 2.1 总体评价

核心玩法闭环完整、体验语言统一,是"能玩明白"的产品。但以 Wordle 类产品的成功要素对照——**每日仪式感、社交攀比、内容稀缺性**——目前只做实了第一项的一半。以下按影响面排序。

### 2.2 P0 问题与方案

#### P0-1 词库容量不足,产品寿命只有约 2 个月

**问题**:主词库 120 条,每日题只从 Lv1-2 共 60 条中选;且 `hashDate(date) % pool.length` 不是排列,存在相邻日期撞题、部分成语永不出现的问题。上线即进入内容枯竭倒计时。

**方案**:
1. 主词库扩到 500+ 条(优先"常见 + 有典故"条目,与现有 TODO#4 合并执行);
2. 每日选题从"哈希取模"改为 **洗牌序列**:用固定种子对题池做一次 Fisher-Yates 洗牌,按 `puzzleNumber % pool.length` 顺序取题(Wordle 原版做法),保证周期内不重复、全覆盖;
3. 建立内容生产管线:每条新成语必须带拼音、部首、部首位置、释义、出处、5 条线索,`project-health` 已有校验框架,补一个"新增词条 lint"脚本即可。

#### P0-2 缺少每日召回机制,留存靠玩家自觉

**问题**:Wordle 类产品命脉是"明天再来"。当前连胜(streak)是唯一钩子,没有订阅消息提醒、没有断签挽回,连胜断了就断了。

**方案**:
1. 接入**微信订阅消息**:结算页顺手请求"明日新题提醒"授权(一次性订阅,结算时机授权率最高),云函数定时触发器每日推送;
2. 增加**补签/护盾**机制:看一条激励视频补回昨日断签(与已规划的广告体系天然契合);
3. 首页对"连胜即将升级"做预告(如"再赢 2 天升举人"),把 童生/秀才/举人 称号体系补全到 进士/状元,并在个人页展示徽章墙。

#### P0-3 分享传播力弱:纯文字卡片,无战绩图

**问题**:`onShare` 的 `imageUrl` 为空,分享卡片是默认截图;Wordle 病毒传播的核心——**无剧透的彩色格子战绩**——只存在于 title 文本里(小程序卡片 title 仅显示一行,emoji 网格实际不可见)。多数页面缺 `onShareTimeline`(朋友圈)。

**方案**:
1. 用离屏 Canvas 生成**战绩分享图**:日期 + 题号 + 彩色格子网格 + 连胜印章,不含答案;
2. 全页面补 `onShareTimeline`;
3. 结算弹层把"分享"做成主按钮 + 生成图预览,当前"找朋友来比试比试"是次级感受;
4. 埋点验证分享率(见 P0-5)。

#### P0-4 双人实时对局不符合微信社交节奏,建议补异步玩法

**问题**:双人局要求**同时在线**,而微信好友互动天然异步(发出去,对方半小时后看到)。房间 2 小时过期 + 需等待对方入席,预期完成率低;这也是当前异常流(TODO#7)复杂的根源。

**方案**:
1. 优先做**异步挑战**:单人提示猜词已有 `?idiom=xxx` 同题分享深链——在此基础上补"发起挑战 → 好友完成后双方收到比分卡片"闭环(云端存一条 challenge 记录即可),开发量远小于实时对局的异常流补全;
2. 实时双人局保留但降级为"次要玩法",异常流按 TODO#7 补齐后再推广;
3. 每日题结算页增加"和好友比今日战绩"(基于 `game_results` 已有数据,以分享回流承载好友关系)。

#### P0-5 数据埋点缺失,上线后无法迭代

**问题**:TODO 里排在 P2,但没有漏斗数据,灰度测试(TODO#13)得到的只有主观反馈。埋点必须先于灰度。

**方案**:上线前接入 `wx.reportEvent`(或自建 `logEvent` 云函数),覆盖最小事件集:`enter_home / start_daily / submit_guess / win / lose / use_hint / watch_ad_done / share_tap / share_open / create_room / join_room`。一天工作量,决定后续所有优先级判断的可信度。

### 2.3 P1 问题与方案

| # | 问题 | 方案 |
|---|---|---|
| P1-1 | **结算页缺"全网对比"时刻**:Wordle 的"今天 38% 的人 3 次猜中"是核心爽点,当前 `game_results` 已有数据但未聚合展示 | 云函数聚合当日尝试次数分布,结算弹层加"你击败了 X% 的玩家"+分布条形图(个人页已有分布组件可复用) |
| P1-2 | **成语学习价值未变现**:答案释义一闪而过,教育向差异化没做实 | 增加「词卡册」:猜过的成语自动收藏,含释义/出处/典故,可翻阅;与"印记"世界观天然契合,也是留存资产 |
| P1-3 | **历史页语义混乱**:云端是"今日排行",本地降级却是"我的历史记录",同一 UI 两种含义 | 拆成两个区块:「今日榜」(仅云端,空态引导)与「我的印记」(本地历史);降级时只展示后者 |
| P1-4 | **首次启动无引导**:玩法说明是被动入口,五色反馈规则(绿/蓝/黄/紫/黑)认知成本高 | 首次进入主局自动弹 3 步引导(已有 guide 弹层素材,加 storage 标记);五色建议精简展示层级——紫色"近音"信息量低,可视觉弱化 |
| P1-5 | **锦囊获得感不足**(已在 TODO#1) | 按既定方向执行:语义方向 + 排除边界;建议再加"消除一个不在答案中的部首格"这类**确定性收益**,比文字提示感知强 |
| P1-6 | **昵称随机生成不持久**:`getPlayerName()` 未设置昵称时每次调用随机返回,同一玩家在房间/排行榜里名字漂移 | 首次生成后写入 storage 固化;进双人房间前引导设置昵称 |

### 2.4 P2 建议

- **成就体系**:首胜、3 天连胜、1 次猜中、零提示破题……徽章墙放个人页(数据本地即可起步);
- **每日题归档**:「往期题目」可回玩(不计连胜),缓解新用户"只有一题可玩"的空虚感;
- **周主题题包**:如"三国周""诗词周",提示猜词按主题组织,制造运营节奏;
- **深浅色适配**与 iOS 安全区细节巡检(部分页面已处理,统一走查)。

## 三、资深软件架构师视角

### 3.1 总体评价

代码可读性好、注释密度得当、降级路径完备,数据同步用脚本 + 健康检查约束是聪明的轻量方案。主要风险集中在:**答案下发到客户端的信任模型、云函数并发一致性、跨端重复逻辑、以及页面级状态机缺乏抽象**。以下按风险排序。

### 3.2 P0 问题与方案

#### A-1 双人局答案对猜词者可见(公平性漏洞)

**问题**:`getRoomState` 云函数原样返回整个 `room`(含 `gameState.currentIdiom`);`room-sync.js` 的 `.watch()` 更是直接监听 `idiom_rooms` 集合——只要客户端有读权限,猜词者抓包或调试即可看到答案。前端仅靠 `me.role === 'hinter' ? currentIdiom : ''` 做展示层隐藏,信任模型是错的。

**方案**:
1. `getRoomState` 按视角脱敏:猜词者返回的 `room.gameState.currentIdiom` 置空,新增 `myView` 字段(服务端已算好 me/opponent,顺势完成);
2. `idiom_rooms` 数据库权限收紧为客户端不可直接读,客户端**统一走云函数轮询**,放弃对该集合的直接 `.watch()`;若要保留 watch,把敏感字段拆到仅提示者可见的独立集合;
3. `idiom_hints_live` 的 watch 无敏感字段,可保留。

#### A-2 每日题与排行榜完全信任客户端(反作弊为零)

**问题**:答案词库整包打进客户端(`data/idioms.js` 含全部答案与释义);`submitResult` 接受客户端上报的 `attempts/won` 原文入库。排行榜数据可任意伪造;`getDailyIdiom` 云函数已存在却未被使用。

**方案**(分两步走,不必一步做重):
1. **上线前(低成本)**:`submitResult` 服务端校验——用 `getDailyIdiom` 同一算法核对 `answerText` 与 `date` 匹配、`attempts ∈ [1,6]`、同一 openid 当日限一条(已有)+ 拒绝未来日期;
2. **中期**:主局改为"云函数判分"模式——客户端提交猜测,云端返回五色结果,答案不再下发;本地词库仅保留"是否成语"校验集。练习模式可继续用本地词库。

#### A-3 云函数读-改-写无事务,并发丢更新

**问题**:`manageRoom`/`submitGuess`/`submitHint` 均为 `get → 内存修改 → update` 模式,且多处**整体重写 `players` 数组**(如 `submitGuess` 计分、`advanceRound` 换角色)。两人同时操作(双方各自点"下一局"、加入与计分并发)会互相覆盖;`advanceRound` 的幂等判断(`lastResult.round !== currentRound`)在并发读下会双双通过,导致跳轮。

**方案**:
1. 关键写路径改用 `db.runTransaction`(云开发原生支持),或在 `update` 条件中携带乐观锁字段(如期望的 `currentRound`),更新失败即重读重试;
2. 玩家分数改为点路径更新或拆出独立字段,避免整数组重写;
3. `advanceRound` 显式幂等:请求携带 `expectRound`,服务端不匹配直接返回当前房间状态(客户端已能处理)。

#### A-4 时区不一致,非东八区用户每日题与服务端错位

**问题**:三处"今天"实现不一致——`utils/daily.getToday()` 用**设备本地时区**,`puzzleNumber` 用固定 +08:00,云函数 `getDailyIdiom` 用 UTC+8。设备时区 ≠ 东八区时,客户端选题与云端(以及未来的全网统计)不一致;还可用改系统时间刷连胜。

**方案**:统一"游戏日"定义为北京时间:`utils/daily.getToday()` 改为 UTC+8 计算(照抄云函数实现);连胜校验中期随 A-2 一并挪到服务端。

### 3.3 P1 问题与方案

#### A-5 跨端重复逻辑,修改需 N 处同步

**问题清单**:
- 计分公式:`idiom-match.js: calcMatchScore` ≡ `submitGuess/index.js: calcScore`(连 `DIFFICULTY_SCORE_ADJUST` 常量都重复,本次逐行比对暂未漂移,但属手工同步);
- `isRoomExpired/getTime/ROOM_TTL_MS`:在 4 个云函数中各复制一份;
- `getToday/getYesterday`:`app.js`、`utils/daily.js`、`index.js: updateStreak` 三份;
- 题池筛选(`defaultEligible` 判定):`idiom-match.js` 与 `manageRoom/index.js` 两份;
- `two-player-game.js` 内部:`watchHints` 重建 + `setData` 重置块出现 3 次。

**方案**:
1. 建 `shared/` 目录存放纯函数模块(计分、房间过期、日期、题池筛选),`sync-data.js` 已有的复制管线扩展为"共享代码同步"(云函数独立部署无法引用包外文件,复制是云开发下的合理方案,但要**由脚本生成而非手工维护**);
2. `two-player-game.js` 抽 `_resetRound(room)` 与 `_rewatchHints(round)` 两个私有方法,消除三段重复;
3. `project-health` 加一条"共享模块派生文件同步"检查,防回退。

#### A-6 无自定义组件,页面间 UI 重复

**问题**:自绘 tab-bar、四格输入(index / idiom-match / two-player-game 三套相似实现)、结算弹层、玩法说明弹层,均为页面内复制。这是"统一历史页/个人页/双人页视觉"(TODO#11)迟迟不动的结构性原因——没有组件,统一视觉 = 改 N 处。

**方案**:按收益排序组件化:`<idiom-input>`(四格 + 隐藏 input + 聚焦管理,一次修 IME 问题三处受益)→ `<result-modal>` → `<app-tabbar>` → `<guide-modal>`。不必一次做完,新需求触达哪个就先抽哪个。

#### A-7 `reLaunch` 模拟 Tab 的代价

**问题**:首页/历史/我的之间跳转全用 `wx.reLaunch`,每次销毁重建全部页面栈:状态丢失、白屏闪动、`onShow` 全量重算。同时 `app.json` 未配置 tabBar,却在每页自绘底栏。

**方案**:二选一并全局统一——
- **方案 a(推荐)**:启用原生 tabBar(首页/印记/我的),自绘底栏移除;需按规范准备 81×81 PNG 图标(与 Logo 重设计 TODO#3 合并执行);Tab 页跳转改 `switchTab`,状态天然保留;
- **方案 b**:保留自绘,但 Tab 间跳转改 `wx.redirectTo`(同层替换,不清栈),仅"从深层页回首页"用 `reLaunch`。

#### A-8 健康检查过度耦合实现细节

**问题**:`project-health.js` 大量断言具体 WXSS 字符串(如 `linear-gradient(135deg, #e45b58 0%, #f0a641 100%)`、具体 padding 值)。每次视觉迭代都要同步改测试,测试从"防回归"退化为"复述实现";且断言口径与 `PROJECT_KNOWLEDGE.md` 描述的配色演进已出现漂移苗头。

**方案**:
1. 分层:引擎/数据/约束类检查(WXML 禁调方法、数据同步、题库质量)保留并加强——这些是真不变量;
2. 视觉类断言改为**语义断言**(如"主 CTA 类存在且被 hero-card 引用"),具体色值交给设计 token:在 `app.wxss` 定义 CSS 变量,健康检查只验证"页面未硬编码绕过 token";
3. 引擎补充**基于行为的单测**:`scoreGuess` 的重复字消耗、`checkPinyin` 边界(ü、轻声)、`getHintPositions` 的 3 位保证——当前回归主要覆盖最后一项。

### 3.4 P2 建议

| # | 问题 | 方案 |
|---|---|---|
| A-9 | `App.onError` 未注册,线上异常黑盒 | app.js 注册 `onError/onUnhandledRejection`,上报到云函数或 `wx.reportEvent`;与产品侧 P0-5 埋点同一管线 |

## 四、落地路线图(建议)

### 第一阶段:上线闸门(1-2 周)

1. A-1 双人局答案脱敏 + 权限收紧(公平性,约半天)
2. A-4 统一游戏日时区(约半天)
3. P0-5 / A-9 最小埋点 + 全局异常上报(约 1 天)
4. A-2 第 1 步:`submitResult` 服务端校验(约半天)
5. A-3 云函数事务/乐观锁改造(1-2 天)
6. 原 TODO#2 真机双人全链路回归 + TODO#3 Logo/图标(并行)
