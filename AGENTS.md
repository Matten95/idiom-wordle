# Idiom Wordle — Codex 开发约束

## 项目概要
- **成语版 Wordle**：每日猜成语，拼音反馈 + 部首提示 + 社交分享
- **微信小程序** 原生框架，云开发模式
- **Node.js 脚本** 用于引擎测试、每日成语轮换

---

## 一、代码修改原则

1. **最小化修改** — 只改当前问题涉及的代码，不顺手重构无关模块
2. **不改动已有正确逻辑** — 修 Bug 时不改变周边正常的数据流和状态管理
3. **零新错误** — 修改后确保编译通过，不引入 WXML/JS/WXSS 语法错误
4. **保持风格一致** — 命名、缩进、注释密度、引号风格完全匹配上下文
5. **只改受影响文件** — 不为"统一风格"去批量修改不相关的文件
6. **先读后改** — 编辑任何文件前必须先使用 Read 工具确认最新内容

---

## 二、小程序专属约束

1. **WXML 模板禁止调用 JS 方法** — `'⭐'.repeat(n)`、`.map()`、`.filter()` 等不能在模板中写，必须在 JS 端预计算后通过 data 绑定
2. **主包体积 ≤ 2MB** — 新增页面/组件前确认不超限
3. **Tab bar 图标规范** — 81×81 像素 PNG，单文件 ≤ 40KB
4. **隐私 API 必须声明** — 若使用 `getFuzzyLocation`、`chooseAddress` 等，必须在 `app.json` 的 `requiredPrivateInfos` 中列出
5. **`setData` 单次 ≤ 256KB** — 大数据必须分片更新
6. **页面栈 ≤ 10 层** — `navigateTo` 超过 8 层应改用 `redirectTo` 或 `reLaunch`
7. **云函数无循环依赖** — 每个云函数独立部署，不互相引用
8. **`wx:key` 必填** — 所有 `wx:for` 必须指定唯一的 `wx:key`
9. **Tab 页面用 `switchTab`** — 不要用 `navigateTo` 跳转 Tab bar 页面
10. **模拟器中 `wx.cloud` 可能不可用** — 关键路径保留 mock 降级逻辑

---

## 三、工作流约束

1. **先读后改** — 修改任何文件前先用 Read 确认内容
2. **改后必验证** — 每次代码修改后通过 MCP 工具检查：
   - `getexceptions` — 查运行时异常
   - `getlogs` — 查控制台错误
   - `screenshot` — 确认 UI 正常
3. **一次只做一件事** — 一个 Task 对应一个功能点，不混搭
4. **复杂改动先 Plan** — 涉及 3 个以上文件或新页面时，必须先进入 Plan 模式，用户确认后再执行
5. **失败先分析** — 报错时先读日志定位根因，不盲目重试或改代码猜测
6. **Task 追踪进度** — 重要改动创建 Task（Create → in_progress → completed）
7. **命名约定** — 页面文件用小写加连字符（`publish-post`），组件文件同规则；变量用 camelCase

---

## 四、禁止操作

- **不要修改 `app.json` 中 tabBar 的 pagePath 顺序** 除非明确要求
- **不要在 WXML 模板中写 JavaScript 表达式**（如方法调用、正则等）
- **不要删除仍在使用的 mock 数据** 在云开发正式切换之前
- **不要同时修改多个页面的业务逻辑** — 聚焦当前任务
- **不要忽略 `wx:if` vs `hidden` 的区别** — 频繁切换用 `hidden`，条件渲染用 `wx:if`

---

## 五、技术栈速查

| 层 | 技术 |
|----|------|
| 视图 | WXML + WXSS（CSS 变量主题） |
| 逻辑 | 原生 JS（ES6+），Page/Component 构造器 |
| 后端 | 微信云开发（云函数 + 云数据库） |
| 引擎 | Node.js（成语匹配、拼音反馈、每日轮换） |
| 类型 | 纯 JavaScript，无 TypeScript |
